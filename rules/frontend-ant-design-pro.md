# Frontend Ant Design Pro Rules

## Scope

- 适用于 Ant Design Pro 管理端页面、请求封装、表单提交、列表请求、枚举和错误处理。
- 接口与类型以后端生成结果为准，前端只做消费和轻量组装。

## Generated APIs

- 接口与类型一律通过 `npm run openapi` 生成。
- 禁止手改生成物：
  - `services/ant-design-pro/*Controller.ts`
  - `typings.d.ts` 里的 API 类型
- 后端接口改动后，先同步生成，再改前端调用。
- 不要在业务代码里手写与生成类型等价的重复接口类型。

## Request Types

- 请求参数、表单提交值优先复用生成的 `API.xxxReq`。
- 字段一致时，表单提交直接透传 `values`。
- `ProTable` 的 `request` 优先直接传 `params`，不要多包一层无意义请求对象。
- 列表返回值优先直接写 `data: res.data?.records ?? []`。
- 避免在页面层补 `as API.Xxx[]` 这类多余断言。

## Forms and State

- 当表单字段与接口 Req 完全一致时，`onFinish` 或提交逻辑直接传 `values`。
- 弹框状态优先用独立布尔值管理，不要为了收口状态而写成枚举切换。
- 页面存在多个弹框时，优先使用清晰的布尔状态名而不是一个大一统状态字段。

## Enums

- 枚举优先由后端接口下发。
- 只有后端没有对应枚举时，才允许前端兜底写死。
- 不要为了“临时能用”就把后端枚举写死在页面里。

## Error Handling

- 用户触发的接口调用需要统一错误提示。
- 页面层如需定制化异常交互，再局部补充处理逻辑。
- 默认不重复写一层冗余 `catch`。
- 如果页面没有明确的特殊兜底诉求，优先交给请求基础层统一处理。

## Permissions

- 一个按钮或操作只绑定一个明确权限点。
- 禁止用多个权限做兜底放宽显示条件。
- 不要为了“看起来能进页面”而混入额外权限判断。

## API Change Flow

- 后端接口变更后，先执行 `npm run openapi`，再改前端业务调用。
- 不要手动改生成物再回头补生成，避免和后端契约不同步。
- 如果生成物与页面写法不匹配，先修生成链路，再修业务代码。
